# Project Instructions: page-tree

このプロジェクトにおける開発の指針とルールです。

## 1. 基本原則

- **日本語での応答**: 全ての対話、ドキュメント、コメント、docstringは日本語で行ってください。
- **docstringの徹底**: 全てのクラス、メソッド、関数には必ずdocstringを記載してください。
- **コメントによる可読性の確保**: 複雑なロジックや処理の流れを理解しやすくするため、適切な箇所に日本語でコメントを記載してください。
- **Ruffの遵守**: コードは常に `ruff` によるチェックをパスする状態を維持してください。
- **uvの活用**: パッケージ管理やスクリプトの実行には `uv` を使用してください。

## 2. コーディング規約

### 2.1 docstring 形式
Google スタイルまたは NumPy スタイルを推奨しますが、プロジェクト内で統一してください。

```python
def fetch_url(url: str) -> str:
    """
    指定されたURLの内容を取得します。

    Args:
        url: 取得対象のURL。

    Returns:
        取得したページの内容。

    Raises:
        HTTPError: リクエストに失敗した場合。
    """
    pass
```

### 2.2 コメント
- 「なぜその処理が必要か」という意図が伝わるように記述してください。
- `ruff` の規則に従い、コードとコメントの間に適切なスペースを入れてください。

### 2.3 型ヒント
- Python 3.14+ の最新機能を活用し、全ての関数に型ヒントを付けてください。

## 3. 技術スタック・実装方針

- **非同期処理**: `httpx` と `asyncio` を使用した非同期通信を基本とします。
- **CLI**: `click` を使用してインターフェースを構築します。
- **エラーハンドリング**: ネットワークエラーやパースエラーに対して、適切な例外処理とユーザーへのフィードバックを行ってください。
- **ロギング**: コアロジック内では `print` ではなく `logging` モジュールを使用してください。

## 4. ワークフロー

- **依存関係の追加**: `uv add <package>` を使用してください。
- **静的解析・リンタ**:
  - `uv run ruff check .` でコードをチェックし、`uv run ruff format .` で整形してください。
  - `uv run mypy .` で静的解析を行ってください。
  - ソースコード（テスト含む）を作成・更新した際は、必ずこれらを実行し、すべてのチェックを通過させてください。
- **テスト**:
  - `pytest` を使用します。新機能を追加した際は、必ずテストを追加してください。
  - テストは、1つの関数に1ケース、または `pytest.mark.parametrize` を用いて複数ケースをまとめて記述してください。
  - パラメータ化したテストには、各ケースに識別可能な `ids` を指定してください。
  - `pytest` は `--log-cli-level=DEBUG` オプションを付けて実行し、エラー時の分析を行ってください。テストパス時も、ログ出力が想定通りであることを確認してください。
- **ファイル操作**:
  - ファイルの作成・更新・修正は、必ず提供されたツール（`write_file`, `replace` など）を使用して行ってください。シェルコマンドによるファイル操作は禁止します。
